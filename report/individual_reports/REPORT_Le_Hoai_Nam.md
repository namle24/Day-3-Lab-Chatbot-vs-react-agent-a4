## II. Debugging Case Study (10 Points)

- **Problem Description**: The agent frequently fell back to simulated logic due to Gemini provider errors (403 access denied and 429 quota exceeded). Additionally, a runtime exception occurred when passing `max_output_tokens` to the Gemini client which didn't accept that argument.

- **Log Source**: Example telemetry and server log entries in `server_rebuild.log` show quota and signature errors:
  - Quota error: `AGENT_ERROR_FALLBACK ... 429 You exceeded your current quota ... model: gemini-2.5-flash`
  - Signature error: `GenerativeModel.generate_content() got an unexpected keyword argument 'max_output_tokens'` ([server_rebuild.log](server_rebuild.log)).

- **Diagnosis**:
  - Quota and permission errors originate from the Gemini project/account and specific model quotas (free tier limits), causing repeated 403/429 responses.
  - The runtime TypeError indicated a mismatch between the wrapper code's expectation (`max_output_tokens` param) and the installed `google.generativeai` client version API.

- **Solution**:
  - Added runtime detection and fallback: if the Gemini call raises quota/permission errors, the service will attempt to instantiate and call `OpenAIProvider` (if `OPENAI_API_KEY` is available) to fulfill the agent's request ([src/services/chat_service.py](src/services/chat_service.py)).
  - Wrapped Gemini `generate_content` calls in a try/except to attempt the parameterized call first and fall back to a no-arg call when the client rejects `max_output_tokens` ([src/core/gemini_provider.py](src/core/gemini_provider.py)).
  - Rebuilt the RAG index and cleared local DB and logs to ensure the knowledge base is up-to-date and reproducible.

## III. Personal Insights: Chatbot vs ReAct (10 Points)

1. **Reasoning**: The ReAct agent structure (Thought → Action → Observation) helped explicitly surface tool usage and intermediate steps, allowing us to call `lookup_vehicle` and `compare_vehicles` deterministically instead of relying on the LLM to implicitly quote snippets.
2. **Reliability**: The agent can be less reliable when the model cannot access tools due to quota/permission errors; in such cases a plain Chatbot fallback (with deterministic DB lookups) performed more predictably.
3. **Observation Influence**: Observations from the tool calls (RAG snippets and structured comparison outputs) significantly improved final answers by providing concrete numeric values (prices, quarters) the LLM could reference.

## IV. Future Improvements (5 Points)

- **Scalability**: Move from a pickle-based RAG index to a persistent vector DB (FAISS/Weaviate) for faster and concurrent-safe lookups.
- **Safety**: Add a supervisor/validator LLM that verifies tool call arguments and sanitizes tool outputs before exposing them to users.
- **Performance**: Stream LLM responses to the UI and persist full agent replies in the DB, allowing the UI to show full text without truncation and letting the LLM maintain token limits.
